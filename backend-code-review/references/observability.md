# 可观测性审查清单

## 检查项

### 8.1 日志

- [ ] 日志是否结构化（JSON 格式，便于日志平台检索）
- [ ] 关键业务流程节点是否有 INFO 日志（订单创建、支付完成等）
- [ ] 可恢复异常是否有 WARN 日志（重试成功、降级触发、慢查询）
- [ ] 影响业务的错误是否有 ERROR 日志（外部服务失败、数据不一致）
- [ ] 日志是否包含 traceID 用于链路关联
- [ ] 日志是否包含关键业务数据（orderID、userID 等）
- [ ] 是否避免在循环中打 INFO 及以上级别日志
- [ ] 敏感信息是否脱敏

### 8.2 指标监控

- [ ] 是否采集了基础指标：QPS、延迟 P50/P95/P99、错误率
- [ ] 是否有关键业务指标（订单量、支付成功率、队列积压深度）
- [ ] 是否有基础设施指标（CPU、内存、磁盘、连接数、goroutine 数）

### 8.3 链路追踪

- [ ] 请求入口是否生成 TraceID
- [ ] TraceID 是否在服务间传递（HTTP Header / gRPC Metadata）
- [ ] 所有日志是否携带 TraceID
- [ ] 是否使用 OpenTelemetry / Jaeger / Zipkin 等标准工具

### 8.4 告警

- [ ] 告警是否可操作（收到后知道该做什么）
- [ ] 是否区分了严重级别：

| 级别 | 响应时效 | 典型场景 |
|------|---------|---------|
| P0 | 立即响应 | 错误率 > 5%、服务不可用、DB 连接耗尽 |
| P1 | 30 分钟 | P99 延迟 > 2s、队列积压 > 10000 |
| P2 | 当日处理 | 磁盘使用率 > 80%、慢查询增多 |
| P3 | 规划处理 | 内存缓慢增长、技术债务 |

- [ ] 告警阈值是否明确（避免"差不多就行"）
- [ ] 是否定期审查并清理长期无人处理的告警
