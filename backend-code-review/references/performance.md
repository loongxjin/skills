# 性能与可扩展性审查清单

## 检查项

### 6.1 性能方法论

- [ ] 优化前是否先量化现状（PProf / APM / 压测获取基准数据）
- [ ] 是否先定位瓶颈再优化（CPU / 内存 / IO / 锁竞争 / 网络）
- [ ] 是否只优化瓶颈，避免过度优化
- [ ] 优化后是否有压测对比验证效果

### 6.2 缓存策略

- [ ] 热点读数据是否使用了缓存
- [ ] 缓存策略是否匹配业务特征：

| 数据特征 | 推荐策略 |
|---------|---------|
| 变更不频繁 | 直接缓存 + 合理 TTL |
| 变更频繁，允许短暂不一致 | Cache Aside（先更新 DB 再删缓存 + 延迟双删） |
| 变更频繁，不允许不一致 | Write Through（同步更新 DB 和缓存） |

- [ ] 是否防范了缓存穿透（查询不存在数据 → 布隆过滤器/缓存空值）
- [ ] 是否防范了缓存雪崩（大量缓存同时过期 → TTL 加随机偏移）
- [ ] 是否防范了缓存击穿（热点 Key 过期 → 互斥锁/永不过期+异步刷新）
- [ ] 缓存是否有容量上限，防止内存无限增长

### 6.3 批量操作与数据量

- [ ] 是否避免了一次性加载全部数据到内存
- [ ] 大批量处理是否分批执行（batch size 建议 500）
- [ ] API 返回数据是否有分页和上限
- [ ] 定时任务处理的数据量增长后，执行时间是否会超过调度间隔
- [ ] 数组/切片是否会无限增长导致 OOM

### 6.4 长期运行考量

- [ ] 是否有内存泄漏风险（goroutine 泄漏、缓存无上限、连接未释放）
- [ ] 连接池是否有最大连接数限制和泄漏检测
- [ ] 文件描述符是否有监控（未关闭的文件/连接/Socket）
- [ ] 是否监控 goroutine 数量趋势（持续增长 = 泄漏）
- [ ] 是否有数据归档策略防止表无限增长
